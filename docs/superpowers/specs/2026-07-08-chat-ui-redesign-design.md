# AI-Brain 对话页交互改造设计文档

> 状态：已对齐，待实现规划
> 日期：2026-07-08
> 范围：对话页 UI / 交互优化（ChatGPT 风格）

---

## 1. 背景与目标

当前对话页已经具备基础聊天能力，但视觉与交互仍偏原型态，主要问题集中在：

- 消息区缺少居中内容列，AI 回复横向过宽，可读性差
- 用户消息与 AI 消息层次弱，气泡样式偏旧
- 侧边栏大量默认标题「新对话」，信息辨识度低
- 输入区呈现为整条底栏，产品感不足

本次改造目标不是新增聊天能力，而是在**不改变核心聊天流程**的前提下，把对话页体验提升到更接近成熟 AI Chat 产品的状态。

明确目标：

1. 让消息阅读更轻松，长回答不再铺满整屏
2. 让消息、空状态、输入区形成明确视觉层级
3. 让会话列表更好用，自动摆脱大量重复的「新对话」
4. 补齐删除会话能力，但避免把范围扩成整套会话管理系统

---

## 2. 范围与边界

### 本次纳入

- 对话页主区域布局重做
- 消息气泡与空状态优化
- 输入框改为底部居中悬浮卡片
- 侧边栏视觉优化
- 会话首条用户消息自动生成标题
- 删除会话能力
- 会话 API 小补充（改标题、删会话）
- Markdown / 引用样式优化

### 本次不做

- 移动端适配
- 侧边栏折叠
- 手动重命名 UI
- 新增 UI 组件库
- 概念实验室页面联动改版
- 多轮 query 改写、聊天能力升级

说明：虽然会补 `PATCH /sessions/{id}` 能力，但本次仅作为数据层预留，不强制增加前端手动重命名入口。

## 3. 设计原则

### 3.1 视觉方向

整体风格采用**接近 ChatGPT 的深色对话产品感**：

- 深色背景保持不变
- 使用居中内容列，而不是整页铺满
- 用更弱的块状边框和更柔和的圆角，减少「管理后台感」
- 通过间距、宽度、透明度建立层次，而不是依赖大量描边

### 3.2 交互原则

- 常用动作优先露出，低频动作 hover 后露出
- 用户聚焦当前对话，不让侧边栏信息抢主视觉
- 新建、切换、删除的行为结果必须直观且可预期
- 保持现有核心流程稳定：新建、切换、发送、流式输出都不能被打断

### 3.3 实现原则

- 优先复用现有组件结构，不引入新依赖
- 前端以现有 `ChatPage`、`MessageList`、`MessageInput`、`SessionSidebar` 为基础重组
- 后端只做最小必要 API 扩展，不改数据库 schema

---

## 4. 当前结构与问题定位

当前对话页由以下部分组成：

- `ChatPage`：整体布局
- `MessageList`：消息列表与空状态
- `MessageInput`：输入框与发送按钮
- `SessionSidebar`：会话列表
- `useSession`：会话状态与切换逻辑

现存问题：

### 4.1 主区域

- `chat-main` 只有最基础的纵向布局，没有形成阅读列
- `message-list` 用整屏宽度渲染，AI 内容块过宽
- 空状态只有单段欢迎语，缺乏引导动作

### 4.2 消息展示

- 通过「你 / AI」纯文本标签区分角色，辨识方式生硬
- AI 消息是大面积深灰矩形，显得厚重
- Markdown 仅覆盖标题、加粗、行内 code，排版细节不够

### 4.3 输入区

- 输入区域是整条底栏，与消息区割裂
- 发送按钮是普通矩形按钮，整体更像后台表单，不像聊天产品

### 4.4 侧边栏

- 列表中大量「新对话」缺乏可区分性
- 时间戳常驻展示，视觉噪声偏大
- 没有删除能力

## 5. 目标形态

```
┌─ 侧边栏 ─┬──────────────────────────────┐
│ 新建对话 │            顶部留白           │
│ 会话列表 │  ┌──── 居中消息列 ────┐      │
│ hover操作│  │ 用户/AI 消息流      │      │
│ 删除会话 │  │ 空状态 / 示例问题    │      │
│          │  └────────────────────┘      │
│          │   ┌── 悬浮输入卡片 ───┐       │
│          │   │ textarea + 发送   │       │
│          │   └───────────────────┘       │
└──────────┴──────────────────────────────┘
```

主区域阅读焦点集中在中间列，侧边栏退居次要信息层。

## 6. 主区域设计

### 6.1 居中消息列

主区域保持全高，但实际消息内容限制在居中窄列中：

- 内容最大宽度：约 `768px`
- 左右 padding：约 `24px`
- 消息列表与输入卡片共享同一最大宽度

这样可以直接解决当前 AI 回复横向过宽的问题，并让用户消息与 AI 消息形成稳定阅读路径。

### 6.2 空状态

空状态从单一文案升级为「欢迎 + 示例问题」：

- 标题：延续 AI-Brain 定位
- 副标题：说明可问课程知识与概念
- 三个示例问题 chip，可点击直接填充并发送

示例问题仅作为引导，不依赖后端新能力。

### 6.3 输入卡片

输入区由底部整条栏改为**底部居中悬浮卡片**：

- 圆角 16px
- 细边框 + 轻微阴影
- textarea 自动增高（1 到 5 行）
- 发送按钮改成更轻量的图标化按钮

输入卡片与消息区宽度对齐，形成统一的中轴线。

---

## 7. 消息展示设计

### 7.1 角色识别

取消当前的纯文本角色标签，改为更自然的消息结构：

- 用户消息：右对齐，紫色系气泡
- AI 消息：左对齐，配一个简洁头像标识

头像只做轻量角色提示，不引入复杂资料卡。

### 7.2 气泡样式

用户消息：

- 紧凑圆角气泡
- 更明显的品牌紫色，但控制发光强度
- 适合短句和连续多条输入

AI 消息：

- 不再使用厚重的大灰块
- 改为更轻的背景或近透明容器
- 强调文本排版和留白，而不是块感

这样可以把视觉焦点从容器边框转移到回答内容本身。

### 7.3 Markdown 与引用

现有轻量 Markdown 继续保留，但要增强样式层次：

- 标题间距更清晰
- 行内 code 更柔和
- 段落行高更舒适
- `[引自: ...]` 或 `[引自：...]` 渲染为淡色引用标签

目标是让课程型回答更像「讲解卡片」，不是一整块挤在灰盒子里。

### 7.4 流式状态

流式生成时不显示生硬的「思考中...」文字块，改为更轻的状态反馈：

- 三点跳动或呼吸动画
- 保持底部自动滚动
- 不影响已有流式拼接逻辑

---

## 8. 侧边栏设计

### 8.1 视觉层级

侧边栏维持固定宽度，但更克制：

- 宽度收敛到约 `220px`
- 选中态改为「浅紫底 + 左侧强调条」
- 默认主展示为标题
- 时间戳降级为 hover 或次级信息

这样可以减少列表噪声，让用户更快扫到会话内容。

### 8.2 自动标题

当会话标题仍为默认值「新对话」时，用户发送首条消息后自动改名。

规则：

- 只对默认标题生效
- 取首条用户消息
- 去除换行与多余空白
- 截断到约 20 个字符，超出加省略号

示例：

- `什么是 Transformer 和 RAG 的关系` → `什么是 Transformer 和 RAG...`

该能力优先在后端写入，保证刷新后仍然存在。

### 8.3 删除会话

每条会话在 hover 时显示删除入口：

- 默认隐藏，hover 出现
- 当前选中项也可见，避免找不到
- 删除前使用轻量确认

删除后行为：

1. 若列表仍有其他会话，切到第一条
2. 若列表为空，自动创建一个空白新对话
3. 若删除的是非当前会话，仅更新列表，不影响当前内容

该策略可以避免用户删完后看到空白死状态。

## 9. 数据与 API 设计

当前后端已有能力：

- `POST /sessions`
- `GET /sessions`
- `GET /sessions/{id}`
- `POST /sessions/{id}/messages`

本次补充两个接口：

| 接口 | 方法 | 用途 |
|------|------|------|
| `/sessions/{id}` | `PATCH` | 更新标题 |
| `/sessions/{id}` | `DELETE` | 删除会话及其消息 |

### 9.1 自动标题写入点

推荐在 `add_message()` 内处理：

- 仅当 `role == "user"` 时参与判断
- 仅当 session 当前标题是 `新对话` 时更新
- 只在首条用户消息到来时触发

好处：

- 标题持久化在同一事务链路附近
- 前端不需要额外判断「第几条用户消息」
- 刷新页面后状态一致

### 9.2 删除逻辑

删除会话时需要一并删除关联消息：

1. 删除 `messages` 中该 `session_id` 的记录
2. 删除 `sessions` 中对应会话

不改 schema，不依赖数据库级联约束。

---

## 10. 前端状态设计

`useSession` 在本次改造后承担的职责：

- 拉取与切换会话
- 保持当前空会话不重复创建
- 支持删除会话后的回退选择
- 在发送首条消息后同步最新 sessions 列表，拿到自动标题

建议补充的方法：

- `removeSession(id)`
- `refreshSessionsAfterSend()` 或在 `handleSend()` 后局部刷新

状态原则：

- 消息流仍以当前 `currentId` 为准
- 删除当前会话时，先决定新的 `currentId`，再清理或重新加载消息
- 避免在流式过程中误触发全量重载导致闪烁

## 12. 验收标准

实现完成后应满足以下标准：

1. 消息阅读区域为居中窄列，AI 回复不再横向铺满主区域
2. 输入区为底部悬浮卡片，输入体验更接近聊天产品
3. 空状态提供可点击示例问题
4. 用户与 AI 消息区分清晰，但不过度依赖文字标签
5. 默认标题会话在首条用户消息后自动变成可识别标题
6. 用户可以删除会话，且删除当前会话后的切换逻辑正确
7. Markdown 与引用信息更易读
8. 现有会话创建、切换、流式输出、历史查看能力不回退
9. 已有「当前已经是空白新对话时不重复创建」逻辑继续成立

## 13. 风险与控制

### 风险 1：发送后立即刷新 sessions 导致界面闪动

控制方式：

- 优先局部更新当前会话标题
- 如需全量刷新，放在发送完成后进行，而不是流式过程中

### 风险 2：删除当前会话后出现空白或错误选中

控制方式：

- 删除前先计算后续要切换的目标会话
- 若无剩余会话，立即创建空白会话兜底

### 风险 3：Markdown 规则增强后误伤普通文本

控制方式：

- 仅对明确的引用格式做轻量增强
- 不引入重型 Markdown 解析器，避免范围失控

## 14. 实施建议

建议按以下顺序实现：

1. 先补后端 title / delete API
2. 再改 `useSession` 的删除与同步逻辑
3. 再做消息区与输入区结构改造
4. 最后做侧边栏样式与 hover 交互

这样可以先把数据行为稳定住，再做视觉层改造，降低联调复杂度。
